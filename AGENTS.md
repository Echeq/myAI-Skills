# myAI-Skills

OpenCode skill packages — 7 prose-only `.md` agents with **no executables, deps,
tests, or CI**. No build, test, lint, or typecheck commands exist — do not attempt
to run any.

Repo-local OpenCode config: `opencode.jsonc` (2 model-aware subagents for orchestrator).

All skills are plain YAML-frontmatter instruction files (`.agents/skills/<name>/SKILL.md`).
The `src/`, `tests/`, `package.json` structure in `docs/guides/creating-skills.md` is
aspirational — none exists today.

## Critical constraints

- **No commands to run.** No `npm test`, `npm run build`, `tsc`, `ruff`, etc.
- **Do not edit `docs/` files directly.** They are generated by `@ai-docs`. Edit
  `.agents/skills/<name>/SKILL.md` instead. The `docs/` tree can drift from the
  skills — do not treat it as authoritative.
- **New skills must be prose-only `.md` (frontmatter + instructions).** The
  `src/`, `tests/`, `package.json` model is aspirational; do not add executable
  code unless the user explicitly requests it.

## Structure

```
opencode.jsonc                   Model-aware subagent definitions (orchestrator-scout, orchestrator-deep)
.agents/
  memory/orchestrator/          Session memory & token tracking (index.md + tokens.md)
  plan/                          Generated plans by date subdirectories
  skills/<name>/SKILL.md         Agent definition (frontmatter + instructions)
  skills/ai-git/                 Git/GitHub skill hub with sub-modules
    SKILL.md                      Router
    commit.md / branch.md / pr.md / release.md
docs/
  README.md                       Skill index
  skills/<name>.md                Per-skill doc page (generated by @ai-docs)
  guides/                         Usage & creation guides
  reference/                      Conventions & ARCHITECTURE.md (ADRs)
  audit/                          @ai-docs audit reports
  auto-report/                    @auto-report generated reports
  ai-audit/                       @ai-audit audit reports
  log/                            @ai-docs --log generated logs (gitignored)
```

subcommand flags listed above.

## Skill SKILL.md format

Frontmatter fields seen across existing skills:

```yaml
---
name: <kebab-case-name>
description: <one-line description>
triggers:
  - "@name"
allowed-tools: Read, Write, Bash, Glob, Grep  # optional
---
```

All skills at minimum use `name` + `description`. Some add `triggers` and `allowed-tools`.

## Generated paths

- `docs/auto-report/edited_*.*` — `@auto-report`
- `docs/ai-audit/AUDIT_REPORT_*.md` — `@ai-audit`
- `docs/audit/DOCS_AUDIT_REPORT.md` — `@ai-docs audit`
- `.agents/memory/orchestrator/` — `@ai-orchestrator` session memory
- `.agents/plan/` — `@ai-orchestrator` generated plans (date-based subdirectories)
- `.agents/memory/ai-audit/` — `@ai-audit` regression tracking

## Conventions (aspirational, not yet enforced)

- **Self-contained skills** — own deps, tests, docs, never share `node_modules`.
- **Config injection** — parameters, never `process.env` globals.
- **SemVer** — every skill starts at `1.0.0`.
- **TypeScript** by default; **Python** for data/scripting.
- **English** for all code, comments, variable names.

## Platform

- **Windows.** Shell commands in skills (e.g. `ai-log-generate`) use PowerShell syntax.
- Source of truth for skill definitions = `.agents/skills/<name>/SKILL.md`.
  The `docs/` tree is derived and should never contradict the skills — but it can
  drift. Run `@ai-docs audit` or `@ai-docs update` to reconcile.
