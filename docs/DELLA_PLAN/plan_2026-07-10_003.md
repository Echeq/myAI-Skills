# Plan: Redesign ROUTER and ORCHESTRATOR as Conversational Agents with Technical Skill Engines

**Generated:** 2026-07-10 | **Model:** deepseek-v4-flash | **Version:** 1.0
**Request:** Create two well-differentiated agents (ROUTER and ORCHESTRATOR) with clear Agent vs Skill separation

## Objective

Redesign ROUTER and ORCHESTRATOR as **conversational agents** (user-facing, decision-makers) backed by **technical skill engines** (called via `skill("name")`). The Agent talks to the user and decides; the Skill executes the heavy logic.

---

## Architecture

```
User
  └── DELLA (strategic planner)
        ├── Decides: ROUTER or ORCHESTRATOR?
        │
        ├── ROUTER Agent (lightweight, daily tasks)
        │     └── Calls skill("ai-router") for pipeline work
        │     └── Calls individual skills directly (ai-audit, ai-docs, etc.)
        │
        └── ORCHESTRATOR Agent (complex, multi-step)
              └── Calls skill("ai-orchestrator") for DAG engine
              └── Auto-routes to skills via capability registry
```

| Layer | What | Files |
|-------|------|-------|
| **Agent** | Conversational, user-facing, makes decisions | `agent/ROUTER.md`, `agent/ORCHESTRATOR.md` |
| **Skill** | Technical engine, called by agents via `skill("name")` | `.agents/skills/ai-router/SKILL.md`, `.agents/skills/ai-orchestrator/SKILL.md` |

---

## Design: ROUTER Agent (`agent/ROUTER.md`)

**Role:** Default lightweight agent for daily tasks. Fast, token-efficient, Flash-first.

### Behaviour

```
User request
  ├── Trivial? (greeting, question, info lookup)
  │     └── Answer directly. Done.
  ├── Simple? (edit 1 file, run 1 command)
  │     └── Do it yourself with tools. Done.
  ├── Medium? (fits a single skill: audit, docs, git)
  │     └── skill("ai-audit"), skill("ai-docs"), etc.
  └── Complex? (needs plan + execute + review)
        └── skill("ai-router") → pipeline
```

### Key traits
- Classifies into 4 tiers: trivial → simple → medium → complex
- Direct answers for trivial/simple (no pipeline, no skill loading)
- Calls individual skills directly for medium tasks
- Only uses `skill("ai-router")` pipeline for complex work
- Minimal memory: only history + current plan

## Design: ROUTER Skill (`.agents/skills/ai-router/SKILL.md`)

**Role:** Technical pipeline engine. Not user-facing. Called by ROUTER Agent.

### Pipeline modes

| Mode | When | Flow |
|------|------|------|
| **quick** | Default for short requests | executor(flash) → review(flash or skip) |
| **plan** | Action verbs or length > 150 | planner(pro) → executor × N → review(pro) → fix loop |
| **debug** | "error", "fail", "bug" in query | read → executor → review(flash) → fix → log |

### Changes from current
- Add frontmatter: `triggers`, `allowed-tools`
- Clarify it is a technical engine, not user-facing
- Keep state paths pointing to `.agents/memory/ai-router/`

## Design: ORCHESTRATOR Agent (`agent/ORCHESTRATOR.md`)

**Role:** Heavy-duty agent for complex multi-step tasks. Strategic, deliberate.

### Behaviour

```
User request (complex)
  ├── Fits a single skill?
  │     └── Route to skill directly (ai-audit, ai-docs, etc.)
  ├── Multi-step but linear?
  │     └── Delegate to ROUTER: skill("ai-router")
  └── Has dependencies / parallel work / cascade risk?
        └── skill("ai-orchestrator") → DAG engine
```

### Key traits
- Clear "when to use me" table vs ROUTER
- Explains DAG, cascade, deadlock in conversational terms
- Routes to specific skills when task fits one domain
- Suggests ROUTER when the task doesn't need a DAG
- Detailed memory: task_states.json, current_plan, history, archived plans

---

## Design: ORCHESTRATOR Skill (`.agents/skills/ai-orchestrator/SKILL.md`)

**Role:** DAG execution engine. Not user-facing. Called by ORCHESTRATOR Agent.

### Core capabilities
- Dynamic classification (structured LLM prompt, not keyword heuristics)
- Capability registry (auto-discovers installed skills)
- 8-state task lifecycle (READY → RUNNING → BLOCKED → COMPLETED → FAILED → CANCELLED → SKIPPED → PAUSED)
- Cascade failure propagation (transitive BFS)
- Deadlock detection (auto-fail BLOCKED tasks with terminal but incomplete deps)
- Task-adaptive review (different criteria per task type)
- DAG engine powered by `dag.py` (stdlib Python CLI)

### Changes from current
- Add frontmatter: `triggers`, `allowed-tools`
- Clarify it is a technical engine, not user-facing
- Keep state paths pointing to `.agents/memory/ai-orchestrator/`
